## Overview: Four Phases

```
PHASE 0   Pre-LOI        Documents only. No infrastructure.
PHASE 1   After LOI      AI Telegram Support Bot.
PHASE 2   After 1 client AI Deployment Automation.
PHASE 3   3+ clients     AIOps Alert Classifier.
```

Each phase gates on the previous. Nothing is built speculatively.

## Phase 3 — AIOps Alert Classifier (3+ clients)

```
INCIDENT DETECTION AND DIAGNOSIS FLOW

  ┌─────────────────────────────────────────────────────────────────┐
  │  UptimeRobot (or custom health check)                           │
  │  Monitors: client.hutabyte.id every 5 minutes                  │
  │  Alert triggers on: HTTP non-200, timeout, SSL expiry           │
  └──────────────────────┬──────────────────────────────────────────┘
                         │ webhook POST
                         ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  aiops/alert_handler.py                                         │
  │  Receives: {monitor_id, status, url, timestamp, check_type}    │
  │  Looks up: client from monitor_id → client config              │
  │  Fetches: Odoo VPS metrics (CPU, RAM, disk, Docker status)     │
  │  Applies PII firewall: only ops/infra data, no business data   │
  └──────────────────────┬──────────────────────────────────────────┘
                         │
                         ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  aiops/classifier.py                                            │
  │                                                                 │
  │  Claude API (Sonnet 4.6):                                       │
  │  system: "You are an Odoo infrastructure incident classifier.   │
  │           Given alert + system metrics, classify the incident   │
  │           and generate a structured summary for the engineer."  │
  │  input: {alert_data, vps_metrics, odoo_logs (non-PII)}         │
  │  output: {                                                      │
  │    category: "disk_full | memory_oom | odoo_crash |            │
  │               db_connection | ssl_expiry | network",           │
  │    severity: "P1 | P2 | P3",                                   │
  │    likely_cause: "string",                                      │
  │    recommended_action: "string",                               │
  │    context: "relevant log excerpt"                             │
  │  }                                                             │
  └──────────────────────┬──────────────────────────────────────────┘
                         │ CLASSIFY ONLY — NO AUTO-REMEDIATION
                         ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  aiops/incident_formatter.py                                    │
  │                                                                 │
  │  Formats structured summary for Hutabyte engineer:             │
  │                                                                 │
  │  🔴 [P1] client.hutabyte.id DOWN                               │
  │  Category: disk_full                                           │
  │  Likely cause: Odoo filestore grew past 95% of 40GB disk       │
  │  Recommended: df -h → find large dirs → clean /opt/odoo/data   │
  │  Time: 2026-05-25 08:14 WIB                                    │
  │  Diagnosis time: < 5 minutes target                            │
  └──────────────────────┬──────────────────────────────────────────┘
                         │ Telegram message to Hutabyte engineer
                         ▼
             Engineer reads summary
             Engineer executes fix MANUALLY
             Engineer marks resolved

  DECISION: NO AUTO-REMEDIATION (locked in D4, CEO review)
  Rationale: AI-caused changes to production databases = data loss risk.
             AI classifies, humans always execute.

AIOPS/ DIRECTORY STRUCTURE:

  aiops/
  ├── alert_handler.py        Webhook receiver + metrics fetcher
  ├── classifier.py           Claude API classification call
  ├── incident_formatter.py   Structured summary generator
  └── runbooks/               Reference context for classifier
      ├── disk_full.md
      ├── memory_oom.md
      ├── odoo_crash.md
      └── ssl_expiry.md
```

## Security Boundary Summary

```
DATA CLASSIFICATION

  ALLOWED in Claude API payloads (whitelist):
  ├── odoo.module_name
  ├── odoo.module_state (installed/uninstalled)
  ├── odoo.latest_version
  ├── odoo.install_date
  ├── odoo.performance_metrics (response_time_ms, query_count)
  └── odoo.error_log (last 100 lines, redacted)

  NEVER in Claude API payloads:
  ├── res.partner (any customer/vendor record)
  ├── hr.employee (any HR record)
  ├── account.move (any fund transaction)
  ├── project.task (content fields — assignee names, descriptions)
  └── Any field containing: name, email, phone, NIK, amount, salary

  ENFORCEMENT:
  ├── Layer 1: odoo_client.py only requests whitelisted fields
  ├── Layer 2: pii_firewall.py rejects non-whitelisted keys
  └── Layer 3: CI gate (.github/workflows/pii-firewall.yml) blocks
               merge if any test payload fails whitelist check

  LEGAL BASIS: UU PDP 2022 — cross-border data transfer to Anthropic
  API limited to operational/config data only. Personal data requires
  explicit consent and separate DPA.
```
