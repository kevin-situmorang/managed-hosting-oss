## Docker Configuration

ERPNext uses the official `frappe_docker` setup maintained by the Frappe team.

### Docker Compose (Simplified)

```yaml
# docker-compose.yml (generated by deploy/playbook_generator.py)
# Based on: https://github.com/frappe/frappe_docker

services:
  backend:
    image: frappe/erpnext:v15
    environment:
      - DB_HOST=db
      - DB_PORT=3306
      - REDIS_CACHE=redis-cache:6379
      - REDIS_QUEUE=redis-queue:6379
    volumes:
      - sites:/home/frappe/frappe-bench/sites
    restart: unless-stopped

  frontend:
    image: frappe/erpnext:v15
    command: nginx-entrypoint.sh
    environment:
      - BACKEND=backend:8000
      - SOCKETIO=websocket:9000
      - UPSTREAM_REAL_IP_ADDRESS=127.0.0.1
    volumes:
      - sites:/home/frappe/frappe-bench/sites
    ports:
      - "8080:8080"
    restart: unless-stopped

  websocket:
    image: frappe/erpnext:v15
    command: node /home/frappe/frappe-bench/apps/frappe/socketio.js
    volumes:
      - sites:/home/frappe/frappe-bench/sites
    restart: unless-stopped

  queue-default:
    image: frappe/erpnext:v15
    command: bench worker --queue default
    volumes:
      - sites:/home/frappe/frappe-bench/sites
    restart: unless-stopped

  scheduler:
    image: frappe/erpnext:v15
    command: bench schedule
    volumes:
      - sites:/home/frappe/frappe-bench/sites
    restart: unless-stopped

  db:
    image: mariadb:10.6
    command: --character-set-server=utf8mb4 --collation-server=utf8mb4_unicode_ci
    environment:
      - MYSQL_ROOT_PASSWORD=${DB_ROOT_PASSWORD}
      - MYSQL_DATABASE=_frappe
    volumes:
      - db-data:/var/lib/mysql
    restart: unless-stopped

  redis-cache:
    image: redis:7-alpine
    restart: unless-stopped

  redis-queue:
    image: redis:7-alpine
    restart: unless-stopped

volumes:
  sites:
  db-data:
```

## Nginx Configuration

ERPNext's frontend container handles its own internal routing. Nginx on the host
terminates SSL and proxies to the frontend container.

```nginx
# /etc/nginx/sites-available/{client-slug}.hutabyte.id
server {
    listen 80;
    server_name {client-slug}.hutabyte.id;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name {client-slug}.hutabyte.id;

    ssl_certificate     /etc/letsencrypt/live/{client-slug}.hutabyte.id/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/{client-slug}.hutabyte.id/privkey.pem;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 120s;
    }

    location /socket.io/ {
        proxy_pass http://127.0.0.1:9000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

## ERPNext API Access (for Support Bot and AIOps)

ERPNext exposes a REST API. The support bot and AIOps components use read-only access.

```python
# Connection details stored in age-encrypted client config
# configs/clients/{client-slug}.enc.yaml
{
  "erp_type": "erpnext",
  "host": "https://{client-slug}.hutabyte.id",
  "site_name": "{client-slug}.hutabyte.id",
  "api_key": "...",       # ERPNext API key (read-only user)
  "api_secret": "...",    # ERPNext API secret
  "erp_type_version": "v15",
  "monitoring_id": "uptimerobot-{monitor-id}"
}
```

### ERPNext Read-Only API User Setup

1. Create a dedicated ERPNext user: `hutabyte-bot@internal.hutabyte.id`
2. Assign role: **System Manager read-only** — or create a custom role with only
   these DocType permissions (read access, no write/create/delete):
   - Installed Applications
   - Error Log
   - Scheduler Log
   - System Settings (read only)
3. Generate API key + secret (User → API Access)

### ERPNext REST Query Methods (required by erp_router.py)

These are the concrete queries `erp_router.py` must implement for ERPNext clients.
Defined here before building erp_router.py, per T9 pre-condition.

| Method | ERPNext REST call | Returns |
|---|---|---|
| `get_installed_apps()` | `GET /api/method/frappe.utils.get_installed_apps` | List of app dicts with name, version |
| `get_error_logs(limit=100)` | `GET /api/resource/Error Log?limit=100&fields=["method","error","creation"]` | Recent error traces |
| `get_scheduler_status()` | `GET /api/resource/Scheduler Log?limit=20&fields=["method","status","scheduled_time"]` | Recent scheduler runs |
| `get_system_health()` | `GET /api/method/frappe.utils.response.build_response` + VPS metrics | System status dict |

**PII whitelist for ERPNext (enforced by pii_firewall.py):**

Allowed keys in ERPNext API response payloads passed to Claude:

```python
ERPNEXT_ALLOWED_FIELDS = {
    # Installed apps
    "app_name", "version", "branch",
    # Error log
    "method", "error", "creation",  # error field is redacted of values
    # Scheduler log
    "status", "scheduled_time",
    # System / infra metrics (VPS-level)
    "cpu_pct", "mem_mb", "disk_pct", "response_time_ms",
    "worker_queue_depth",
    # Generic
    "name", "idx",  # ERPNext internal record identifiers (not person names)
}
```

**Forbidden DocTypes — never queried:**
`Customer`, `Supplier`, `Employee`, `Salary Slip`, `GL Entry`, `Sales Invoice`,
`Purchase Invoice`, `Payment Entry`, `Stock Entry`, `User` (content fields),
`Address`, `Contact`

## Backup Strategy for ERPNext

ERPNext backup differs from Odoo. Both the MariaDB database AND the sites volume
must be backed up.

```bash
# MariaDB dump (runs daily via backup Ansible role)
mysqldump -h db -u root -p${DB_ROOT_PASSWORD} ${client-slug}-prod \
  | gzip > /backup/{client-slug}-$(date +%Y%m%d).sql.gz

# Sites volume backup (includes file attachments, custom configs)
tar -czf /backup/{client-slug}-sites-$(date +%Y%m%d).tar.gz \
  /home/frappe/frappe-bench/sites/{client-slug}.hutabyte.id/

# Upload both to S3-compatible storage
s3cmd put /backup/{client-slug}-*.gz s3://hutabyte-backups/{client-slug}/
```

The backup restore verification (T14) must test restoring BOTH the MariaDB dump
AND the sites volume to confirm the site is fully operational.
