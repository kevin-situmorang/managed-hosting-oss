## Problem Statement

Indonesian SMEs (>20 employees) are paying Rp 170K–500K/user/month for SaaS business
systems (ERP, CRM, project management, HR). They cannot self-host open-source
alternatives because they have no internal IT team — no infrastructure, no DevOps, no
one to maintain servers. Hutabyte's thesis: the open-source software is mature and
free; the missing piece is a managed ops layer (hosting, implementation, security
patching, support) that costs far less than per-seat SaaS licensing.

The core value prop is not software discounts — it is "the IT team you don't have."

## Demand Evidence

- 2 pilot users: one retailer company, one non-profit organization (both >20 employees,
  no internal IT team)
- Both willing to try open-source version under a free pilot — confirmed interest only
- Current SaaS spend: Rp 170K–500K/user/month (est. Rp 3.4M–10M/month for 20 users)
- Neither has confirmed willingness to pay — pilot is the demand validation vehicle
- Target pricing: Rp 500K/month flat (unlimited users) + Rp 1M/month/app optional
  24/7 support add-on

NOTE: Interest is not demand. A signed LOI at a real price is the only output that
matters from the pilot phase.

## Status Quo

- No internal IT/DevOps team at either pilot company
- Current software stack: not yet mapped (mapping is homework before provisioning)
- Non-profit use cases: program/project monitoring, fund management, procurement, HR
  management, consolidated reporting
- Retailer use cases: not yet mapped (likely POS, inventory, sales — different stack)
- Pain: operational fragmentation, high per-seat cost, zero IT ownership

## Target User & Narrowest Wedge

Primary target for this phase: the non-profit pilot company.

Reason: their use case list maps cleanly to Odoo 17 Community out of the box. The
retailer needs a different app configuration — serving both simultaneously destroys
standardization.

The specific human who feels the pain: the Operations Manager or G&A Manager — not
the CEO. This person loses hours weekly to fragmented systems and manual consolidation.
They have not yet been interviewed.

Narrowest wedge: one Odoo 17 Community deployment for the non-profit, pre-configured
for their four stated use cases, hosted on Hetzner VPS, branded as a Hutabyte product,
with a one-page service agreement showing a real monthly price. Goal: signed or rejected
LOI within 4 weeks. Indonesian SME sales cycles typically require 1–2 follow-up meetings
before a signature; plan for this, not a single-meeting close.

## Constraints

- No internal IT team at clients (the value prop, not a constraint to route around)
- Hutabyte must maintain a standardized stack — custom per-client work must be exceptional
  and billable, not the default. Decision rule: any Odoo module not in the standard stack
  requires written approval from the Hutabyte lead and a separate quoted engagement.
- Pricing at Rp 500K/month flat may not cover COGS — must calculate floor before signing
- One segment first: retailer and non-profit need different Odoo module configurations
- No product code exists yet — this is a pre-product business design exercise

## Premises (all agreed)

1. Standardization is the model. One stack, one config, deployed repeatably across all
   clients. Custom per-client work is billable and exceptional.

2. Base price may not cover COGS. Rp 500K/month may not cover hosting + patching +
   support hours at steady state. Real revenue must come from add-ons or the base price
   must be higher. Calculate COGS before setting price.

3. Pick one segment. Retailer and non-profit need different open-source app configs.
   Serve one first, build the playbook, then expand.

4. No demand evidence yet. Two willing test subjects is not demand. A signed LOI at a
   real price is the evidence.

## COGS Worksheet (Calculate Before Setting Price)

| Cost Item | Est. Monthly (IDR) | Notes |
|---|---|---|
| Hetzner VPS CX31 (Singapore) | ~Rp 350K | 4 vCPU 8GB RAM, single-tenant per client for pilot phase |
| Backup storage (S3-compatible) | ~Rp 50K | Daily backups, 30-day retention |
| Monitoring (UptimeRobot free tier) | Rp 0 | Upgrade to paid if SLA is guaranteed |
| Security patching labor (est. 1hr/client/month) | Rp 150K–500K | Depends on Hutabyte engineer rate |
| Support labor (base tier, est. 0.5hr/client/month) | Rp 75K–250K | Scales with client count |
| Total COGS estimate | Rp 625K–1.15M | Per client per month |

Implication: Rp 500K/month base price does not cover estimated COGS. Options:
A) Raise base price to Rp 1M–1.5M/month (recommended — simplest, honest)
B) Subsidize base with mandatory add-on bundling (adds sales complexity)
C) Multi-tenant Odoo to reduce per-client infra cost (DO NOT pursue for pilot —
   multi-tenant Odoo requires separate architecture design: data isolation, upgrade
   sequencing, module conflict management. Out of scope until 10+ clients.)

Decision required before LOI price is named: choose A or B, calculate resulting
price floor, confirm Hutabyte can sustain it.

## Service Agreement Required Clauses

The one-page service agreement must cover at minimum:
1. Monthly fee and payment terms (invoice date, late payment penalty)
2. SLA: uptime target (e.g., 99.5% excluding maintenance windows) and incident response time
3. Data ownership: client owns their data; Hutabyte has no right to share or sell it
4. Backup policy: daily backups, 30-day retention, client can request data export at any time
5. Customization policy: standard Odoo modules included; custom modules billed separately
6. Termination: 30-day written notice, data export provided within 7 days of termination
7. Hosting location: Singapore (Hetzner) — note any data residency implications for
   non-profits handling donor or grant data under Indonesian law

## Approaches Considered

### Approach A: One Vertical, One Client, Real Price (CHOSEN)
Deploy Odoo 17 Community for the non-profit pilot only. Standard config for fund
management, HR, procurement, reporting. Charge a real price. Get a signed LOI or a
clear rejection within 4 weeks.
Effort: S | Risk: Low | Reuses: Odoo Community, Docker Compose, Hetzner VPS
Exit criteria: if no LOI decision by week 4, activate Approach C pricing test.

### Approach B: Two Verticals, Standardized Engine
Separate standard stacks for non-profit (Odoo) and retailer (ERPNext/Odoo POS).
Repeatable deployment pipeline (Ansible or Coolify). Tiered pricing defined upfront.
Effort: M | Risk: Med | Skipped — premature before demand evidence exists.

### Approach C: Sell "Your IT Department," Not Software
Pitch: "Hutabyte is your IT department — fixed monthly fee, we handle everything."
Price at Rp 2M–3M/month. Open-source stack becomes an implementation detail.
Effort: S | Risk: Med
Trigger: activate if pilot rejects price and states any flat fee feels too high, OR
if week-4 LOI gate from Approach A is not met.

## Recommended Approach

Approach A. The only thing that matters before any further investment is demand evidence
with a real price attached. One vertical, one client, one LOI.

If non-profit signs: validate unit economics (COGS vs revenue), then expand to retailer
vertical with a separate standard stack.

If non-profit rejects: find out exactly why. Wrong price, wrong product, or wrong client
are each different answers requiring different responses. If price is the objection, test
Approach C framing immediately.

## AI Enhancement Layer (Added by /plan-ceo-review 2026-05-25)

### Decision: Full AI-Ops Stack — Build After LOI

AI enhancement is confirmed viable and directly addresses the COGS problem.
All three components are designed but NOT built until the LOI is signed.

### Component 1: AI Customer Support Bot

Interface: WhatsApp Business API (primary), web chat (secondary)
LLM: Claude API (Sonnet 4.6 with prompt caching for cost reduction)
Language: Bahasa Indonesia — system prompt, responses, all copy
Architecture: RAG over Odoo 17 docs + Hutabyte runbooks + client module config
Data scope (STRICT — never exceed): Odoo config, module list, error logs,
  performance metrics. NEVER: employee records, donor data, fund transactions,
  HR records. Any query requiring personal records → escalate to human.
Escalation: Unresolved after 2 turns → route to Hutabyte human on WhatsApp
Target metric: >70% of queries resolved without human escalation

### Component 2: AI Deployment Automation

Trigger: New client onboarding
Flow: Intake form (use cases) → Claude API generates Ansible playbook +
  Docker Compose config → Hutabyte engineer reviews → dry-run
  (ansible-playbook --check) → deploy → automated health check
Human gate: Engineer approves before any real provisioning
Target metric: Time-to-live-Odoo < 30 minutes from approved intake

### Component 3: AIOps Monitor + Alert Classifier (Classify Only)

Trigger: UptimeRobot webhook or custom health check failure
Flow: Alert → Claude API classifies and generates incident summary with
  context → Hutabyte engineer receives structured summary → engineer
  executes fix manually
Decision (D4): NO auto-remediation — AI classifies only, humans always execute
Rationale: Eliminates risk of AI-caused data loss on production databases
Target metric: Mean-time-to-diagnosis < 5 minutes after alert fires

### AI Security Constraints (Binding — from D3)

1. Support bot: Odoo read-only API scoped to non-PII operational data only
2. Service agreement must explicitly state AI processing scope and data categories
3. PII firewall test: automated CI check that asserts zero PII fields in
   Claude API payloads before every support bot deployment
4. UU PDP 2022 compliance: cross-border data transfer to Anthropic API limited
   to operational/config data only; no personal data without explicit consent

### Build Sequence (from D5)

Phase 0 (Now): Document AI design, add data handling clause to service agreement
Phase 1 (After LOI): Build AI support bot (~1 week with Claude Code)
Phase 2 (After LOI + first client stable): Build deployment automation (~2-3 weeks)
Phase 3 (After 3+ clients): Build AIOps alert classifier (~1-2 weeks)

|---|---|---|---|
| Support labor | Rp 75K-250K/client/mo | Rp 20K-75K/client/mo | 70% reduction via bot |
| Deploy/patch labor | Rp 150K-500K/client/mo | Rp 50K-150K/client/mo | 60% reduction via automation |

## Engineering Architecture (Added by /plan-eng-review 2026-05-25)

### Decisions Locked

| Decision | Choice | Rationale |
|---|---|---|
| Phase scope | Phase 0+1 deep, Phase 2-3 sketch | Actionable before LOI; revisit before Phase 2 build |
| Bot deployment | Central Hutabyte server | Fix once, applies to all clients |
| RAG approach | Prompt-stuffing (Phase 1) | 200k context fits all modules; upgrade to vector RAG at 5+ clients |
| Bot channel | Telegram first, WhatsApp later | Zero business verification, free, faster to validate |
| Client credentials | age-encrypted YAML, two custodians | Scales to 20+ clients; survives engineer turnover |
| PII firewall | Field whitelist (enforced at Odoo client + payload layers) | Blocks unknown PII by default; auditable by non-engineers |
| Language + repo | Python monorepo (support-bot/, deploy/, aiops/, common/) | Shared PII firewall + Odoo client; Ansible is Python-native |
| Webhook handling | FastAPI async: 200 OK in <1s, reply via sendMessage async | Telegram 5s timeout; Claude API 2-4s; sync = duplicate messages |
| Conversation state | PostgreSQL | Concurrent workers safe; no migration needed later |
| Key management | Two custodians, encrypted backup in private repo | Lose age key = lose all client configs |

### Failure Modes

| Codepath | Production failure | Test coverage | Error handling | User experience |
|---|---|---|---|---|
| Odoo API unreachable | Connection timeout during query | GAP | GAP | Silent failure unless added |
| PII whitelist stale after Odoo upgrade | New field passes through to Claude | CI gate catches at deploy | Whitelist check at query + payload layers | Bot returns PII to user |
| Escalation notification fails | Hutabyte Telegram unreachable | GAP | GAP | User told nothing, case lost |
| age key lost | All client configs unrecoverable | N/A | Two-custodian procedure | Complete service outage |
| Telegram webhook retry (5s timeout) | Duplicate message + confused turn count | GAP | Async handling prevents most cases | Duplicate reply to user |

**Critical gaps:** Odoo timeout handling + escalation failure fallback — both need error handling and tests before first client demo.
